While I'm not a game developer, I am an embedded Software Engineer so I can speak to this a little bit. The way he described the CK2 system sounds pretty par for the course: One primary thread to handle the game state, and one rendering thread to handle GPU rendering. Additional threads are spawned by each individual task as required, and most processing is performance in a sequential manner commanded by the primary game thread. While individual systems may be made parallel within themselves, you still have large sections of code that are sequential in nature and limit how much performance you can squeeze out of your program (Amdhals Law). The CK3 approach is more modern; instead of updating each system in a sequential manner you make each system parallel. For example, instead of doing UpdateSchemes followed by UpdateOpinions, you do both at the same time. Any processing that requires thread locking is done at the end of all other processing. This gives you higher thread utilization, but is less deterministic and can lead to some *very* hard to track down processing bugs if not implemented correctly. The first method has been around for decades; the primary reasons it's still used is due to a combination of baked in support in most off the shelf game engines, and the time/money cost of updating existing engines to use a more modern methodology. You are seeing a general move toward the second however, largely due to per-core CPU performance stalling out in favor of higher core counts (EG: the performance loss is starting to be a large enough problem where it's getting attention).

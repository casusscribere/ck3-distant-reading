# CK3 Distant Reading — NEH Workshop Week 4

Distant reading of the Crusader Kings III dev diaries (245 diaries, ~625k words)
and their community reply threads (~48.9k replies, ~3.2M words), published as a
live GitHub Pages site. See README.md for the full project description.

## Publishing workflow (IMPORTANT)

This repository IS the live site. GitHub Pages serves the `main` branch from the
repository root. **After any change to site content, commit and push to `main`
to deploy — do this automatically, without being asked, whenever site-visible
files change.**

```
git add -A
git commit -m "<describe the change>"
git push
```

- Remote: `casusscribere/ck3-distant-reading` (GitHub CLI is installed and authed)
- Live URL: http://www.kirklundblade.com/ck3-distant-reading/ (custom domain from
  the account-level Pages config — not casusscribere.github.io)
- Deploys land ~1 minute after push; verify with
  `gh api repos/casusscribere/ck3-distant-reading/pages/builds/latest`
- **Never delete `.nojekyll`** — without it, Pages runs the corpus through
  Jekyll, which hangs/fails on CK3 game syntax in the 500+ markdown files
- Site-visible files: `index.html`, everything under `ck3_distant_reading/`,
  and anything else linked from the landing page

## Site structure

- `index.html` — landing page (CK3 parchment theme, links the three reports)
- `ck3_distant_reading/` — the three HTML reports plus their PNG/SVG/JSON assets
- `ck3-style-guide/` — CK3 visual theme tokens (`theme.css`); the landing page
  inlines a subset of these, so keep the two in sync if the palette changes

## Corpus and tooling

- `ck3_dev_diaries/` — dev-diary corpus (`dd_*.md`) + `MANIFEST.md` + capture scripts
- `ck3_dev_diary_replies/` — reply corpus (`rt_*.md`) + `MANIFEST.md`
- `ck3_reply_scraper/` — reply capture/extraction pipeline (Claude-in-Chrome based)
- `distant-reader/` — the reusable distant-reader skill (scripts, report template,
  methodology references); this is what built the corpus and reports
- `salter_best_practices.md` — the methodological standard this project is held to

## Conventions

- Corpus markdown files carry provenance YAML front-matter and inline
  `<!-- artifact:* -->` comments — never strip either; the "flag but do not
  alter" discipline is the point of the corpus.
- Reports are build artifacts: regenerate via `distant-reader/scripts/` rather
  than hand-editing analysis numbers or visualizations in the HTML.
- Analysis follows the Salter eight-step method with Underwood-style
  verification — frequency claims must be ground-truthed against passages
  before they go in a report.
- Manifests (`MANIFEST.md`) are generated by the `build_*_manifest.py` scripts;
  regenerate rather than hand-edit when corpus files change.
